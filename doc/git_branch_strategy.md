## Git Branch Strategy

저희 팀은 5인 규모의 소규모 팀으로서, 불필요한 복잡성은 줄이고 개발 효율성을 극대화하기 위해,
Git Flow 기반의 중앙 리포지토리 전략을 채택했습니다.

---

<br>

### Git Strategy

안정적인 배포와 효율적인 기능 개발을 분리하기 위해 표준적인 Git Flow 를 따릅니다.

<br>

### [ Git Flow ]
안정적인 배포와 효율적인 기능 개발을 분리하기 위해 표준적인 **Git Flow**를 따릅니다.

- **`master`**: 실제 사용자에게 서빙되는 배포용 브랜치입니다. 이 브랜치에 코드가 병합되면 CD 파이프라인을 통해 자동 배포됩니다.
- **`dev`**: 개발 중인 코드가 모이는 통합 브랜치입니다. Feature 브랜치가 병합될 때마다 CI가 동작하여 코드의 정합성을 검증한 후, 배포 테스트를 통해 사용자에게 서빙이 가능한지 테스트합니다.
- **`feat`**: 각 기능 단위로 독립적으로 작업하는 브랜치입니다. 작업이 완료되면 Pull Request를 통해 `dev` 브랜치로 병합됩니다.

<br>

### [ Why Not FORK ]
일반적인 오픈소스 프로젝트의 Fork & Pull 방식 대신에, 단일 리포지토리 방식을 선택한 이유는 다음과 같습니다.

1. **복잡성 제거 & 동기화 비용 절감**
  -  Fork 방식은 원본 저장소와 개인 저장소 간의 잦은 동기화가 필요합니다.
  - 5명의 팀원이 긴밀하게 협업하는 환경에서는 이러한 절차가 오히려 개발 속도를 저해할 수 있다고 판단하였습니다.

2. **효율적인 CI/CD 구축**
  - dev 브랜치에 병합 시 코드 검사와 배포 검사가 이루어지는 파이프라인을 일관되게 관리함으로써 안정적인 배포 환경을 구축하고자 하였습니다.

3. **가시적인 협업 흐름**
  - 모든 팀원들이 하나의 리포지토리 내에서 작업하게 함으로써 이슈 트래킹 및 코드 리뷰 과정이 더 직관적이게 이루어지도록 하였습니다.

<br>

결론적으로 저희는 **불필요한 복잡성을 줄이고, 투명한 협업과 안정적인 배포 환경 구축을 위해서**

**Git Flow 기반의 브랜치 전략을 선택**했습니다.
