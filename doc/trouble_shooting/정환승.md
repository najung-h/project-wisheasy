# DB & Backend 담당자, 정환승입니다.

_**"지하철 역사를 그래프로 구조화하여, 에스컬레이터 중심 안내를 위한 '데이터 토대’를 직접 설계·구현"**_

<br>

## 개요

- **기술 스택:** Python, Django, MySQL, SQLite, Docker, GitHub
- **핵심 목표:** 지하철 역사 내 에스컬레이터 중심 길안내 로직 구현 및 이를 뒷받침하는 도메인/ERD 설계
- **담당 역할:** 지하철 도메인 모델링, 그래프 기반 길안내 로직, DB 스키마 설계 및 CSV 적재를 담당

초기에는 제공된 공공 데이터의 간접적인 역사내 정보만으로 가공하여 길안내를 구현하려 했지만,  
데이터 구조의 불완전성 때문에 “알고리즘은 짰는데 신뢰할 수 없는 서비스”라는 한계에 부딪혔습니다.  

> 도메인 모델링 → 그래프 구조화 → ERD 재설계 → 안내 로직 고도화 → 배포 파이프라인 정비

까지 한 싸이클을 직접 재구축 및 경험하며, 안정적이고 정확한 길안내가 가능한 구조로 발전시키는 데 집중했습니다.

<br>

## 1. 문제 정의와 접근 방향

### 1-1. 공공 데이터 한계 → “필요한 데이터를 직접 설계하기”

- **고민:**  
  제공된 공공 데이터는 ‘길안내’를 위해 설계된 데이터가 아니어서,
  - 역 내부 구조가 불명확하고
  - 출구/대합실/승강장 관계가 직접적으로 표현되지 않아  
    **“길안내 로직을 신뢰할 수 없다”**는 문제가 반복되었습니다.
- **접근:**  
  기존 데이터에 로직을 끼워 맞추는 Bottom-up 방식 대신,  
  **도메인 주도 설계(DDD) 관점에서 “길안내에 진짜 필요한 데이터 구조”를 먼저 정의**하고,  
  거기에 맞춰 데이터를 구축하는 Top-down 방식으로 전환했습니다.

### 1-2. 비정형 지하철 구조 → 출구/대합실/승강장으로 표준화

- **고민:**  
  지하철 역은 각 지형과 환경에 맞춰 설계되어,  
  - 대합실이 여러 개이거나
  - 승강장–대합실–환승 통로 구조가 제각각이라  
  역마다 로직을 따로 짜야 하는 상황이 발생했습니다.
- **접근:**  
  모든 역을 **`출구 – 대합실 – 승강장` 세 가지 공통 구조로 추상화**해 표준화했습니다.  
  “현실은 복잡하지만, 모델 안에서는 단순하게”를 목표로,  
  어떤 역이든 이 세 요소 조합으로 표현 가능하도록 도메인을 정리했습니다.

### 1-3. 알고리즘 중심 DB → 도메인 중심 DB로 재설계

- **고민:**  
  처음에는 BFS를 쉽게 돌리기 위한 구조로만 테이블을 설계해서,
  - DB가 실제 도메인 모델이 아니라 **알고리즘 캐시**처럼 동작했고
  - 로직이 바뀌면 DB 스키마까지 매번 수정해야 하는 구조였습니다.
- **접근:**  
  `Station`, `Line` 같은 **실제 엔터티 중심으로 ERD를 재구성**하고,  
  `Node`, `Edge`는 이 도메인을 참조하는 구조로 변경했습니다.  
  최단 경로 계산은 **애플리케이션 레이어에서 동적으로 생성**하도록 분리해  
  데이터 구조와 로직의 결합도를 낮췄습니다.

<br>

## 2. 기술적 고민과 해결 과정

### 2-1. 그래프 기반 길안내 설계

- **고민:**  
  단순 BFS 경로만으로는
  - “에스컬레이터를 최대한 활용하게 한다”
  - “환승/다음 역 방향에 맞는 탑승구를 안내한다”  
  같은 **사용자 목적 기반 길안내**가 어려웠습니다.
- **해결:**  
  - 역 내부 구조를 `Node(출구/대합실/승강장)`와 `Edge(이동 경로)`로 그래프 모델링
  - `Edge`에 실제 운영 중인 에스컬레이터 정보를 연결해  
    **경로 안내 속에 “어떤 에스컬레이터를 지나가는지”를 함께 전달할 수 있도록 설계**
  - “다음 역 방향/환승 여부”를 이용해
    - 다음 역 도착 노드로 환승 여부를 판별하고
    - 다음 역 출발 노드로 방면을 매핑해  
      **사용자 목적(환승/하차역 기준)에 맞는 탑승구 안내**가 가능하도록 로직을 구성

### 2-2. ERD 정규화와 데이터 무결성 확보

- **고민:**  
  초기 ERD에서는 `Station`, `Line`, `Node`, `Edge` 등 여러 테이블에  
  역 이름/호선 정보가 반복 저장되어,
  - 데이터 입력이 번거롭고
  - 중복 업데이트에서 무결성이 깨질 위험이 컸습니다.
- **해결:**  
  - 역·호선 정보를 단일 테이블로 일원화
  - FK(외래키) 사용으로 관계를 명시  
  → 데이터 중복을 제거하고, **향후 다른 기능(편의시설, 혼잡도 등)이 확장되기 쉬운 구조**로 정리했습니다.

### 2-3. 기능과 테이블 구조의 단일 책임 분리

- **고민:**  
  테이블을 알고리즘에 맞게 설계하다 보니,
  - 특정 기능(길안내)에만 종속된 테이블이 생기고
  - 역 편의시설 같은 새로운 기능을 넣으려면  
    **비슷한 데이터를 다시 입력해야 하는 구조**가 되었습니다.
- **해결:**  
  - “테이블은 도메인을 표현하고, 기능은 테이블 관계를 활용해 구현한다”는 원칙으로 전환
  - **단일 책임 원칙(SRP)** 관점에서 테이블 설계를 다시 보며,
    - 데이터 레이어는 기능에 중립적
    - 기능/로직은 애플리케이션에서 조합  
    되도록 구조를 개선했습니다.

## 3. 확장된 프로젝트 경험을 통해 얻은 인사이트

쉽길 프로젝트에서는 단순히 지하철 길안내 기능 하나의 구현으로 끝난 게 아니라,  
**도메인 이해 → 데이터/ERD 설계 → 알고리즘 구현 → 배포/운영까지를** 경험할 수 있었습니다.  
이 과정에서 몇 가지 관점을 얻었습니다.

### 3-1. “기술”보다 먼저, 문제를 어떻게 정의할 것인가

- 처음에는 “주어진 공공 데이터로 알고리즘을 활용해 길안내를 하면 되겠지”라고 생각했지만,  
  실제로 구현해보니 **데이터 자체가 길안내용이 아니라는 근본적인 한계**에 계속 부딪혔습니다.
- 이때 단순히 알고리즘이나 코드만 손보는 게 아니라,
  > “우리가 풀려는 문제(지하철 내 이동 안내)를 위해  
  > 무엇을 ‘정보/데이터’로 정의해야 하지?”  
  를 다시 질문하게 되었습니다.
  “어떤 모델을 쓸까?”보다 먼저  
  **“문제를 어떻게 정의하고, 무엇을 관측 가능한 데이터로 볼 것인가”**가 더 중요하다는 걸 실감했습니다.

### 3-2. 설계에는 항상 트레이드오프가 있다는 것

- 지하철 역 구조를 `출구–대합실–승강장`으로 단순화하는 과정에서는  
  현실의 복잡한 구조를 어느 정도 버리는 대신,
  - 데이터 관리 용이성
  - 로직의 일관성
  - 확장성  
  을 얻는 선택을 했습니다.
- 반대로, 에스컬레이터 안내나 탑승구 안내처럼 세밀한 기능을 넣을 때는  
  다시 세분화된 정보가 필요했고,  
  이때 **“얼마나 쪼갤지, 어디까지 추상화할지”**를 계속 조정해야 했습니다.
- 이 경험을 통해,  
  **“정교함 vs 단순함 vs 운영 부담” 사이에서 트레이드오프를 선택하는 감각**이 중요하다는 걸 배웠습니다.

### 3-3. 기능 · 데이터 · 운영이 따로 노는 게 아니라는 것

- 로직만 맞으면 끝이 아니라,
  - 어떤 스키마로 저장되고,
  - CSV를 어떻게 적재하고,
  - 마이그레이션 히스토리가 배포 파이프라인에서 어떻게 해석되는지까지  
    연결해서 보지 않으면 실제 서비스에서는 문제가 터진다는 걸 경험했습니다.
- 특히 로컬에서는 잘 되는데 배포 환경에서 깨졌을 때,  
  GitHub Actions 로그를 따라가며  
  “코드/DB/데이터/워크플로우가 각각 어떤 역할을 하는지”를 다시 정리하게 됐습니다.
- 앞으로 데이터 파이프라인이나 ML 시스템을 설계할 때도,  
  **“모델 한 덩어리”가 아니라 서비스 전체 흐름 속에서  
  어떤 책임을 지는지까지 같이 봐야 한다**는 감각을 얻었습니다.

### 3-4. 한 역할이지만, 전체 구조를 이해하려는 태도

- 백엔드/DB 파트를 맡았지만,
  - 프론트에서 어떤 값이 들어오고,
  - 그 값이 어떻게 가공돼서 DB에 쌓이고,
  - 나중에 이 데이터가 다시 어떤 기능에 쓰이는지까지 함께 보려고 했습니다.
- 이 과정에서 느낀점은,
  > “내 파트만 잘하면 된다”가 아니라  
  > **“내 파트가 전체 흐름에서 어떤 의미를 가지는지 이해하는 사람”이  
  > AI/데이터 역할에서도 더 강점을 가진다**는 점이었습니다.

## 4.트러블슈팅 

### 4-1. 배포에서 발생한 오류 디버깅
 **상황:**  
  - 로컬에서는 새 ERD + 로직이 잘 돌아가는데
  - 배포 후 서버에서는 스키마 변경이 반영되지 않는 문제 발생
- **원인 추적 과정:**
  1. GitHub Actions 로그에서 `No migrations to apply` 메시지 확인
  2. ERD를 크게 변경하면서 **기존 마이그레이션 파일을 삭제한 것이 문제**였음을 파악  
     → 배포 파이프라인 입장에서는 “이미 0001이 적용된 상태”로 인식
  3. 마이그레이션 히스토리가 중요하다는 것을 체감하고,
     - 이후에는 마이그레이션을 삭제하지 않고
     - 변경 과정을 누적 기록하는 방향으로 수정

### 4-2. CI/CD 자동화 워크플로우(yml)

- **상황 인지**
  - Django 관리 커맨드(`load_csv.py`)를 만들고
  - 로컬에서 `python manage.py load_csv`로 CSV 적재까지 정상 동작 확인
  - 하지만 배포 서버에서는 코드 배포 후에도 **테이블이 비어 있는 상태가 지속**

- **원인 파악**
  - 배포 과정이 **CI/CD 자동화 워크플로우**에 의해 진행되고 있고  
    이 파이프라인에는 `manage.py migrate`까지만 정의되어 있으며  
    **데이터 적재 단계가 별도로 존재하지 않는다는 점**을 확인
  - 즉, 배포 서버 입장에서는 “코드를 올려라”는 지시만 있을 뿐  
    “데이터를 적재하라”는 명시적인 단계가 빠져 있었던 문제

- **해결 및 정리**
  - 워크플로우 yml에 `python manage.py load_csv` 스텝을 추가하여  
    `코드 배포 → 마이그레이션 → CSV 적재`까지 하나의 파이프라인으로 정의
  - 이후 배포부터는 별도 수동 작업 없이도 **코드와 데이터가 항상 동일한 상태로 배포**되도록 구조화

## 5. 개인적인 소감 

이번 프로젝트를 통해 **서비스 개발 프로젝트의 전체적인 흐름을 직접 경험해보자**라는 초기 목표로 시작했는데 충분히 달성했다고 느낍니다.  
기능 구현 과정에서는 초반에 시행착오도 많았지만, 끝까지 붙잡고 개선한 덕분에 만족스러운 결과물을 낼 수 있어 뿌듯했습니다.

특히 원활한 소통과 문서화를 바탕으로 협업의 가치를 느낄 수 있었던 프로젝트라고 생각합니다.
다양한 전공을 가진 팀원들 모두가 본인의 역할에서 공부한 내용, 해결한 이슈, 고민 과정 등을 구체적으로 문서화해서 공유했고,  
매일 스크럼 회의를 통해 “오늘 한 일 · 막힌 부분 · 내일 할 일”을 꾸준히 정리했습니다.  
이런 문서화와 소통이 프로젝트 완성에 있어 가장 큰 핵심이었다고 느낍니다.

프로젝트 마무리를 위해 열심히 노력해준 모든 팀원들에게 감사하다고 전하고 싶습니다.
