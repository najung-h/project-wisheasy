
# 기술적 고난

## 1. PRG 패턴과 세션 상태머신으로 새로고침·뒤로가기 문제 해결

지하철 길찾기 화면은 하나의 폼에서 출발·도착역을 입력하고 그 결과로 여러 단계의 안내 스텝을 보여주는 구조였습니다.  
초기에는 **POST 요청 직후 바로 템플릿을 렌더링하는 방식** 때문에, 새로고침과 뒤로가기 시 안내 단계가 꼬이는 문제가 반복적으로 발생했습니다.

이를 해결하기 위해 다음과 같이 구조를 전면 재설계했습니다.

- `journeys:route`라는 단일 URL에서 `GET` / `POST`를 모두 처리
- 모든 `POST` 요청은 `redirect("journeys:route")`로 끝내는 **PRG(Post-Redirect-Get) 패턴 도입**
- 여정 상태를 표현하는 **세션 전용 네임스페이스** 설계

세션에는 다음과 같은 정보를 저장하도록 했습니다.

- 여정 ID
- 생성 시각
- 안내 스텝 리스트
- 현재 인덱스
- 출발/도착역·출구 정보

이 값들은 헬퍼 함수를 통해 상태 전이를 캡슐화했습니다.

또한 홈 화면의 길찾기 버튼을 항상 `?new=1` 쿼리로 진입하도록 만들어, 이전 여정 상태를 명시적으로 초기화한 뒤 새 여정을 시작하도록 했습니다.  
그 결과 새로고침이나 뒤로가기를 하더라도 **현재 안내 단계가 안정적으로 유지되는 길안내 플로우**를 구현할 수 있었습니다.

## 2. 도메인 전용 예외로 경로 생성 실패를 일관된 에러 UX로 리팩터링

경로 생성 서비스의 특성상 다음과 같은 다양한 실패 케이스가 존재합니다.

- 역명이 잘못 입력된 경우
- 그래프 상에서 최단 경로를 찾지 못한 경우
- 내부 그래프 데이터가 불완전한 경우

초기 구현에서는 다음과 같은 문제가 있었습니다.

- 안내 스텝 리스트 안에 `[오류] 역명 확인 필요` 같은 문장을 섞어서 반환
- 내부 Exception 메시지를 거의 그대로 사용자에게 노출
- 도메인 로직과 오류 표현이 뒤섞여, 정상 스텝과 오류 상태를 코드 레벨에서 구분하기 어려움

이를 해결하기 위해 다음과 같은 리팩터링을 진행했습니다.

- `RouteBuildError`, `GuidanceBuildError`와 같은 **도메인 전용 예외 타입 정의**
- 경로 그래프 생성 과정과 안내 문장 생성 과정에서 실패 시  
  더 이상 오류 스텝을 리스트에 추가하지 않고, 의미 있는 예외를 직접 발생
- 뷰 레이어에서 해당 예외를 구분 처리하여, 사용자에게는 다음과 같이 **단일 안내 문구만 노출**

  > 입력하신 역명으로는 경로를 찾을 수 없습니다. 역 이름을 다시 확인해 주세요.

- 예기치 못한 일반 예외의 경우 내부 메시지는 숨기고,  
  `일시적인 오류가 발생했습니다. 잠시 후 다시 시도해 주세요.`와 같은 일반 안내만 노출

이 과정을 통해 도메인 로직, 에러 처리, 사용자 경험을 분리하면서도  
장애 상황을 더 일관되고 안전하게 관리할 수 있게 되었습니다.

## 3. 역·출구 입력 정규화와 프론트·백엔드 검증 통일

사용자가 입력하는 출구 정보는 다음처럼 매우 다양한 표현을 가질 수 있습니다.

- `1`
- `1번`
- `1 번`
- `1번 출구`
- `1번출구`

실제 DB에는 특정 형식만 저장되어 있어 다음과 같은 문제가 있었습니다.

- 유효성 검사 로직이 복잡해짐
- 화면마다 처리 방식이 달라질 위험 존재

이를 해소하기 위해 다음과 같은 정규화·검증 전략을 설계했습니다.

### 백엔드

- 출구 문자열 정규화 함수 설계  
  - `1`, `1번`, `1번 출구`를 모두 내부적으로 `1번출구` 형태로 통일
- 역별 출구 목록과 비교 시, 공백 등을 제거한 정규화 값 기준으로 비교
- 출구가 비어 있는 경우 대표 출구(예: `1번출구`)를 기본값으로 시도
- 존재하지 않는 출구일 경우, 해당 역의 최대 출구 번호 정보를 기반으로  
  보다 구체적인 에러 메시지 반환

예시 메시지:

- `3번 출구는 존재하지 않습니다. 2번 출구까지 있습니다.`

### 프론트엔드

- 백엔드와 동일한 정규화 규칙과 유효성 검사를 수행하는 JavaScript 함수 추가
- 폼 제출 전에 다음 항목을 미리 검증

  - 출발/도착역 필수 입력
  - 출발역과 도착역이 동일하지 않은지 확인
  - 출구 형식(숫자, `1번`, `1번출구` 등) 검사 및 정규화

이를 통해 사용자는 자연스러운 표현으로 자유롭게 입력하되,  
시스템 내부에서는 일관된 형식으로 안전하게 처리하는 구조를 만들 수 있었습니다.

## 4. 환경별 세션·쿠키 설정 분리로 개발 편의성과 보안의 균형

로컬 개발 환경과 운영 환경은 세션·쿠키에 대한 요구사항이 다릅니다.

- 로컬: 빠른 테스트와 디버깅이 중요
- 운영: HTTPS 기반의 보안이 중요

공통 설정만 사용할 경우 다음 문제가 발생할 수 있습니다.

- 개발 중에는 불필요한 제약 증가
- 운영 환경에서는 보안 옵션 누락 위험

이를 방지하기 위해 환경별 설정을 분리했습니다.

### 로컬 환경

- HTTP 기반 개발 서버를 사용하므로, 다음 옵션을 비활성화

  - `SESSION_COOKIE_SECURE = False`
  - `CSRF_COOKIE_SECURE = False`

- 대신 다음 옵션을 사용해 세션 정리를 쉽게 함

  - `SESSION_EXPIRE_AT_BROWSER_CLOSE = True`  
    (브라우저 종료 시 세션 만료)

### 운영 환경

- HTTPS 전제를 기준으로 secure 쿠키와 적절한 만료 정책 적용
- CSRF, 세션 쿠키 모두 `Secure` 옵션을 활성화하여 보안 강화

이를 통해 개발 단계에서는 인증·세션 관련 기능을 빠르게 검증하면서도,  
실제 서비스 환경에서는 보안을 해치지 않는 구성을 유지할 수 있었습니다.

# 핵심 가치

> 백엔드를 중심으로 하되, 필요할 때 프론트와 서버 운영까지 따라가서 문제를 끝까지 해결하는 개발자

이번 프로젝트에서 저는 주로 백엔드 도메인 모델링과 API 구현을 담당했지만,  
API가 실제 화면과 서버 환경에서 어떻게 동작하는지까지 확인하는 것을 중요하게 생각했습니다.

예를 들어 역 편의시설 조회 기능의 경우 다음과 같이 작업했습니다.

- `/api/stations/<station_id>/facilities/?line_id=<line_id>` 형태의 API 설계
- `FacilityLoc` 모델을 기반으로 역명, 노선명, 편의시설명을 함께 내려주는 직렬화 구성
- 프론트엔드에서 바로 사용할 수 있는 JSON 구조 제공

배포 환경에서 이 API가 빈 배열만 반환하는 문제가 발생했을 때에는 다음과 같이 접근했습니다.

1. 단순히 코드만 의심하지 않고, 로컬에서는 정상인데 prod에서는 비정상인 상황에 주목
2. EC2에 직접 접속하여 Docker 컨테이너 내부에서 Django shell 실행
3. `FacilityLoc.objects.count()`로 운영 DB의 실제 레코드 수 확인
4. CSV 적재 스크립트가 facilityLoc 데이터를 넣지 못하고 있었음을 확인
5. `manage.py load_csv`를 직접 실행해 운영 DB에 데이터를 적재

프론트엔드 측면에서도 다음과 같은 개선을 수행했습니다.

- `route.js`에서 사용되지 않는 `keydown` 핸들러 제거  
  (콘솔 에러 원인 제거)
- 자동완성 함수에서 특정 화면에는 존재하지 않는 DOM 요소 접근에  
  가드 코드를 추가하여 화면 전환 시 불필요한 오류를 방지

이처럼 제 역할의 중심은 백엔드였지만,  
API가 실제로 어떻게 보이고 동작하는지, 서버 환경에서 어떤 데이터가 오가는지까지 확인하며  
문제를 끝까지 추적하는 태도를 핵심 가치로 두고 프로젝트에 참여했습니다.

# 목표

> 지하철 역사 내 에스컬레이터를 최대한 활용하게 해 주는 길안내 서비스를, 백엔드 입장에서 끝까지 책임져 보는 것

서비스의 목표는 복잡한 지하철 역사 구조 속에서  
사용자가 에스컬레이터를 중심으로 이동할 수 있도록 돕는 것입니다.

출발역·도착역·출구·편의시설 데이터를 조합하여, 단순 최단 거리뿐 아니라  
**어디서 어떤 편의시설을 거쳐 올라가야 하는지**까지 안내하는 데 초점을 맞추었습니다.

개인적인 목표는, 백엔드 신입 개발자로서 이 도메인을 한 사이클로 경험해 보는 것이었습니다.

제가 맡아 진행한 주요 작업은 다음과 같습니다.

- 역·노선·출구·편의시설·여정에 대한 도메인 모델링
- 역 편의시설 조회 API 및 길안내 관련 API 설계와 구현
- 세션과 PRG 패턴을 이용한 화면 플로우 안정화
- 프론트엔드에서 axios로 API를 호출하고 자동완성·검색 로직과 연결되는 부분 일부 수정
- EC2 + Docker 환경에서 컨테이너 상태를 확인하고,  
  Django shell과 관리 커맨드로 운영 DB를 진단·수정

이를 통해 **상용 서비스에 가까운 작은 백엔드**를 직접 운영해 보는 경험을 목표로 삼았습니다.

# 기술 스택

## Backend

- Python 3 / Django
- Django REST framework (DRF)  
  - 역 검색 및 편의시설 조회 API  
  - 여정 관련 API 구현
- Django Allauth  
  - Google 소셜 로그인 연동  
  - 로그인 플로우 일부 커스터마이징

## Frontend (기여 범위)

- HTML5, CSS3
- Vanilla JavaScript
- 사용되지 않는 `keydown` 이벤트 핸들러 제거  
  키 입력 시 발생하던 콘솔 에러 제거
- 자동완성 초기화 함수에서 특정 화면에 존재하지 않는 DOM 요소 접근에 대한 가드 코드 추가  
  안내 화면 전환 시 불필요한 에러 방지

## Infra & DevOps (기여 범위)

- AWS EC2 (Ubuntu)  
  - 기존 세팅된 서버에 SSH 접속하여 상태 진단
- Docker / docker-compose  
  - `docker compose -f docker-compose.prod.yml ps/exec` 등을 이용해  
    웹·DB 컨테이너 상태 확인
- Django 관리 커맨드  
  - `manage.py shell`, `manage.py load_csv`를 prod 설정으로 실행  
  - `FacilityLoc.objects.count()`로 운영 DB 상태 확인
- `facility.csv`, `facilityLoc.csv`를 실제 운영 DB에 적재
- 기존 배포 스크립트(`.csv_loaded` 파일을 기준으로 CSV 적재를 스킵하는 로직)에 대한 이해  
  및 개선 아이디어 도출

인프라 전반을 설계하거나 운영을 전담한 것은 아니지만,  
기존 파이프라인과 환경을 이해한 뒤 필요한 범위 내에서 직접 진단·조치하는 경험을 했습니다.

# 소감

이번 프로젝트에서 가장 큰 수확은 **맡은 영역을 넘어서 원인을 끝까지 확인해 본 경험**입니다.

배포 환경에서 역 편의시설 API가 에러도 없이 빈 배열만 반환하던 이슈를 예로 들 수 있습니다.

1. 처음에는 뷰나 serializer를 의심했지만, 로컬에서는 정상 동작함을 확인했습니다.
2. 이후 “그렇다면 prod DB 데이터가 다를 수 있다”는 가설을 세웠습니다.
3. EC2에 접속해 Docker 컨테이너 내부에서 Django shell을 열고 실제 레코드 개수를 확인했습니다.
4. prod 환경의 `FacilityLoc` 데이터가 비어 있고,  
   배포 스크립트에서 CSV 적재를 한 번만 실행하도록 만든 `/data/.csv_loaded` 파일 때문에  
   이후에 추가한 `facility.csv`, `facilityLoc.csv` 적재가 자동으로 스킵되고 있음을 확인했습니다.
5. 수동으로 `load_csv`를 실행한 뒤 다시 API를 호출했을 때,  
   실제 JSON 데이터가 내려오는 것을 확인했습니다.

프론트엔드 측면에서도 다음과 같은 점을 배웠습니다.

- 작은 콘솔 에러라도 남겨 두면 디버깅 시 중요한 신호를 놓치기 쉽다.
- 함께 작업하는 팀원들에게도 불편함을 줄 수 있다.

따라서 백엔드 로직만 구현하는 개발자가 아니라, API·프론트·서버 환경이 어떻게 연결되는지 이해하고  
문제가 발생했을 때 어느 지점까지는 스스로 내려가 볼 수 있는 개발자가 되고자 합니다.

앞으로는 이번 경험을 바탕으로 더 복잡한 도메인과 트래픽이 많은 환경에서도,  
같은 태도로 문제를 정의하고 추적해 나갈 수 있는 백엔드 개발자로 성장하고 싶습니다.
