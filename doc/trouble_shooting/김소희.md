# 기술적 고난 
**① PRG 패턴과 세션 상태머신으로 새로고침/뒤로가기 문제를 해결하였습니다.**
지하철 길찾기 화면은 하나의 폼에서 출발·도착역을 입력하고 그 결과로 여러 단계의 안내 스텝을 보여주는 구조였는데, 초기에는 POST 요청 직후 바로 템플릿을 렌더링하는 방식 때문에 새로고침과 뒤로가기 시 안내 단계가 꼬이는 문제가 반복적으로 발생하였습니다. 이 문제를 해결하기 위해 HTTP 요청 흐름을 전면적으로 재정리하고, journeys:route라는 단일 URL에서 GET/POST를 모두 처리하는 구조로 변경한 뒤, 모든 POST 요청을 redirect("journeys:route")로 끝내는 PRG(Post-Redirect-Get) 패턴을 도입하였습니다. 동시에 여정 상태를 표현하는 세션 전용 네임스페이스를 설계하여 여정 ID, 생성 시각, 안내 스텝 리스트, 현재 인덱스, 출발/도착역·출구 정보 등을 세션에 저장하고, 헬퍼 함수를 통해 상태 전이를 캡슐화하였습니다. 홈 화면의 “길찾기” 버튼은 항상 ?new=1 쿼리로 진입하도록 만들어 이전 여정 상태를 명시적으로 초기화하고 새 여정을 시작하게 하였으며, 그 결과 새로고침이나 뒤로가기에도 현재 안내 단계가 안정적으로 유지되는 길안내 플로우를 구현할 수 있었습니다.

**② 도메인 전용 예외를 설계하여 경로 생성 실패를 일관된 에러 UX로 리팩터링하였습니다.**
경로를 생성하는 서비스 특성상 역명이 잘못 입력되거나 그래프 상에서 최단 경로를 찾지 못하는 등 다양한 실패 케이스가 존재하였고, 이를 일관된 방식으로 처리하기 위해 도메인 전용 예외 설계와 에러 UX 리팩터링을 진행하였습니다. 초기에 구현했을 때는 “[오류] 역명 확인 필요”와 같은 문장을 안내 스텝 리스트 안에 섞어 넣거나, 내부 Exception 메시지를 거의 그대로 사용자에게 노출하는 방식에 의존하고 있어 도메인 로직과 오류 표현이 뒤섞이고, 코드 레벨에서도 정상적인 안내 스텝과 오류 상태를 구분하기 어려운 문제가 있었습니다. 이를 해결하기 위해 RouteBuildError, GuidanceBuildError와 같은 도메인 전용 예외 타입을 정의하고, 경로 그래프 생성 과정과 안내 문장 생성 과정에서 실패가 발생하면 리스트에 “오류 스텝”을 추가하는 대신 의미 있는 예외를 직접 발생시키도록 서비스를 리팩터링하였습니다. 뷰 레이어에서는 이 예외들을 구분해서 처리하여, 사용자에게는 “입력하신 역명으로는 경로를 찾을 수 없습니다. 역 이름을 다시 확인해 주세요.”와 같은 단일 안내 문구만 노출하고, 예기치 못한 일반 예외에 대해서는 내부 메시지를 숨긴 채 일시적인 오류 안내를 보여주도록 UX를 정리하였습니다. 이를 통해 도메인 로직, 에러 처리, 사용자 경험을 분리하면서도 장애 상황을 더 일관되고 안전하게 관리할 수 있게 하였습니다.

**③ 역/출구 입력을 정규화하고 프론트·백엔드 검증을 통일하여 입력 신뢰성을 높였습니다.**
사용자가 입력하는 출구 정보의 표현을 정규화하고 프론트엔드와 백엔드의 검증 규칙을 통일함으로써, 다양한 형태의 입력을 허용하면서도 내부적으로는 일관된 데이터를 다룰 수 있도록 개선하였습니다. 사용자는 "1", "1번", "1 번", "1번 출구", "1번출구" 등 매우 다양한 방식으로 출구를 입력하였지만, 실제 DB에는 특정 형식으로만 저장되어 있어 유효성 검사 로직이 복잡해지고 화면마다 처리 방식이 달라질 위험이 있었습니다. 이를 해소하기 위해 백엔드에서 출구 문자열을 정규화하는 함수를 설계하여 "1", "1번", "1번 출구"를 모두 내부적으로 "1번출구"와 같은 통일된 형태로 변환한 뒤 역별 출구 목록과 비교하도록 검증 로직을 재작성하였습니다. 출구가 비어 있는 경우에는 기본값으로 대표 출구를 시도하고, 존재하지 않는 출구일 경우 해당 역의 최대 출구 번호 정보를 기반으로 보다 구체적인 에러 메시지를 반환하도록 설계하였습니다. 동시에 프론트엔드에도 동일한 정규화 규칙과 유효성 검사를 수행하는 자바스크립트 함수를 추가하여, 출발/도착역 필수 체크, 동일 역 방지, 출구 형식 검증을 폼 제출 전에 미리 수행하게 하였습니다. 이로써 사용자는 자연스러운 표현으로 자유롭게 입력하되, 시스템 내부에서는 일관된 형식으로 안전하게 처리하는 구조를 만들 수 있었습니다.

**④ 환경별 세션·쿠키 설정을 분리하여 개발 편의성과 보안의 균형을 맞추었습니다.**
로컬 개발 환경과 운영 환경에서 요구되는 세션·쿠키 조건이 서로 다르기 때문에, 환경별 설정을 분리하여 개발 편의성과 보안 요구사항의 균형을 맞추고자 하였습니다. 공통 설정만 사용할 경우 개발 중에는 불필요한 제약이 생기고, 반대로 운영 환경에서는 보안 옵션이 누락될 위험이 있어 이를 명확히 나누는 작업이 필요했습니다. 로컬 환경에서는 HTTP로 서버를 띄우는 만큼 SESSION_COOKIE_SECURE, CSRF_COOKIE_SECURE와 같은 옵션을 비활성화하여 브라우저에서 쿠키가 차단되지 않도록 하였으며, 대신 SESSION_EXPIRE_AT_BROWSER_CLOSE = True를 사용하여 브라우저 종료 시 세션이 만료되도록 설정해 개발 중 세션 정리를 쉽게 하였습니다. 반대로 운영 환경에서는 HTTPS를 전제로 secure 쿠키와 적절한 만료 정책을 적용하여 보안을 강화하였습니다. 이러한 구성을 통해 인증·세션 관련 기능을 개발 단계에서 빠르게 검증하면서도, 실제 서비스 환경에서는 보안을 해치지 않는 구성을 유지할 수 있도록 하였습니다.

# 핵심 가치 
**백엔드를 중심으로 하되, 필요할 때 프론트와 서버 운영까지 따라가서 문제를 끝까지 해결하는 개발자**를 지향하였습니다.
이번 프로젝트에서 저는 주로 백엔드 도메인 모델링과 API 구현을 담당하였지만, API가 실제 화면과 서버 환경에서 어떻게 동작하는지까지 확인하고 싶었습니다. 역 편의시설 조회 기능의 경우 /api/stations/<station_id>/facilities/?line_id=<line_id> 형태의 API를 설계하고, FacilityLoc 모델을 기반으로 역명, 노선명, 편의시설명을 함께 내려주는 직렬화를 구성하여 프론트엔드에서 바로 사용할 수 있도록 하였습니다.

또한 이 API가 배포 환경에서 빈 배열만 반환하는 문제가 발생했을 때, 단순히 코드만 의심하는 데서 그치지 않고 EC2에 직접 접속하여 Docker 컨테이너 내부에서 Django shell로 DB 상태를 조회하고, CSV 적재 스크립트가 실제로는 facilityLoc 데이터를 넣지 못하고 있었음을 확인한 뒤 수동으로 manage.py load_csv를 실행하여 문제를 해결하였습니다. 프론트엔드 측면에서도, 제가 전체를 담당하지는 않았지만 route.js에서 사용되지 않는 keydown 핸들러가 콘솔 에러를 유발하는 부분을 제거하고, 자동완성 함수에서 특정 화면에는 존재하지 않는 DOM 요소에 접근하던 로직에 가드 코드를 추가하여 화면 전환 시 불필요한 에러가 발생하지 않도록 수정하였습니다.

이처럼 제 역할의 중심은 백엔드였지만, API가 실제로 어떻게 보이고 동작하는지, 서버 환경에서 어떤 데이터가 오가고 있는지까지 확인하며 문제를 끝까지 추적하는 태도를 핵심 가치로 두고 프로젝트에 참여하였습니다.

# 목표 
**지하철 역사 내 에스컬레이터를 최대한 활용하게 해 주는 길안내 서비스를, 백엔드 입장에서 끝까지 책임져 보는 것**을 목표로 하였습니다.
서비스 자체의 목표는 사용자가 복잡한 지하철 역사 구조 속에서 에스컬레이터를 중심으로 이동할 수 있도록 돕는 것이었습니다. 출발역·도착역·출구·편의시설 데이터를 조합하여, 단순히 최단 거리만이 아니라 **“어디서 어떤 편의시설을 거쳐 올라가야 하는지”**까지 안내하는 데에 초점을 맞추었습니다.

개인적인 목표는, 백엔드 신입 개발자로서 이 도메인을 한 사이클로 경험해 보는 것이었습니다.

- 역·노선·출구·편의시설·여정에 대한 도메인 모델링
- 역 편의시설 조회 API 및 길안내 관련 API 설계와 구현
- 세션과 PRG 패턴을 이용한 화면 플로우 안정화
- 프론트엔드에서 axios로 API를 호출하고 자동완성/검색 로직과 연결되는 부분 일부 수정
- EC2 + Docker 환경에서 컨테이너 상태를 확인하고, Django shell과 관리 커맨드로 prod DB를 진단·수정

까지를 직접 다뤄 보면서, **상용 서비스에 가까운 작은 백엔드**를 운영해 보는 것을 목표로 삼았습니다.

# 기술스택
**Backend**
- Python 3 / Django
- Django REST framework (DRF) – 역 검색 및 편의시설 조회 API, 여정 관련 API 구현
- Django Allauth – Google 소셜 로그인 연동 및 로그인 플로우 일부 커스터마이징

**Frontend (기여 범위)**
- HTML5, CSS3
- Vanilla JavaScript
- 사용되지 않는 keydown 이벤트 핸들러 제거로, 키를 누를 때마다 발생하던 콘솔 에러 제거
- 자동완성 초기화 함수에서 특정 화면에는 존재하지 않는 suggestionsId 요소 접근에 대한 가드 코드 추가로, 안내 화면 전환 시 불필요한 에러 방지

**Infra & DevOps (기여 범위)**
- AWS EC2 (Ubuntu) – 기존 세팅된 서버에 SSH 접속하여 상태 진단
- Docker / docker-compose – docker compose -f docker-compose.prod.yml ps/exec 등을 이용해 웹/DB 컨테이너 상태 확인
- Django 관리 커맨드 – manage.py shell, manage.py load_csv를 prod 설정으로 실행하여 FacilityLoc.objects.count() 조회로 prod DB 상태 확인
- facility.csv, facilityLoc.csv를 실제 운영 DB에 적재
- 기존 배포 스크립트(.csv_loaded 파일을 기준으로 CSV 적재를 스킵하는 로직)에 대한 이해 및 개선 아이디어 도출

인프라 전반을 설계하거나 운영을 전담한 것은 아니지만, 기존 파이프라인과 환경을 이해한 뒤 필요한 범위 내에서 직접 진단·조치하는 경험을 했습니다.

**소감**
내가 맡은 영역을 넘어서 원인을 끝까지 확인해 본 경험이 가장 큰 수확이었습니다.
이번 프로젝트에서 가장 인상 깊었던 순간은, 배포 환경에서 역 편의시설 API가 에러도 없이 빈 배열만 반환하던 이슈를 추적해 나갔던 과정입니다. 처음에는 뷰나 serializer를 의심했지만, 로컬에서는 정상 동작한다는 점을 확인한 뒤 곧바로 “그렇다면 prod DB 데이터가 다를 수 있다”는 가설을 세우고 EC2에 접속해 Docker 컨테이너 내부에서 Django shell을 열어 실제 레코드 개수를 확인하였습니다. 그 결과, prod 환경의 FacilityLoc 데이터가 아예 비어 있고, 배포 스크립트에서 CSV 적재를 한 번만 실행하도록 만든 /data/.csv_loaded 파일 때문에 이후에 추가한 facility.csv, facilityLoc.csv 적재가 자동으로 스킵되고 있었다는 것을 알게 되었습니다. 이후 수동으로 load_csv를 실행하고, 다시 API를 호출했을 때 실제 JSON 데이터가 내려오는 것을 확인하면서 “문제를 끝까지 따라가 본다”는 감각을 얻을 수 있었습니다.

프론트엔드 쪽에서도 제가 전체를 맡은 것은 아니지만, route.js에서 키보드 입력만으로 콘솔 에러가 발생하던 부분이나, 특정 화면에는 존재하지 않는 DOM 요소를 전제로 동작하던 자동완성 초기화 로직을 직접 수정해 보면서 내가 수정한 코드가 실제 사용자 화면에 어떤 영향을 주는지”를 체감할 수 있었습니다. 특히, 작은 콘솔 에러라도 남겨 두면 디버깅 시 중요한 신호를 놓치게 되고, 협업 중인 팀원들에게도 불편함을 줄 수 있다는 점을 몸으로 배웠습니다.

전체적으로 이번 프로젝트를 통해, 백엔드 로직만 구현하는 개발자가 아니라, API·프론트·서버 환경이 어떻게 연결되는지 이해하고 문제가 발생했을 때 어느 지점까지는 스스로 내려가 볼 수 있는 개발자가 되고 싶다는 목표가 생겼습니다. 앞으로는 이번 경험을 바탕으로 더 복잡한 도메인과 트래픽이 많은 환경에서도, 같은 태도로 문제를 정의하고 추적해 나갈 수 있는 백엔드 개발자로 성장하고자 합니다.
